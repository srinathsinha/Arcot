# 1. OVERALL BACKEND GOAL

Build a Node.js/Express backend that performs a **real autonomous compliance workflow** using:

### Real integrations
- **Coinbase CDP Server Wallet**  
- **Real on-chain micro-payment** (testnet)  
- **Authentic x402 invoice/payment flow structure**

### Simulated integrations
- **Locus-style compliance tool** that:
  - Returns **402 invoices**  
  - Validates x402 payments  
  - Returns compliance verdicts (sender clean, receiver darknet hit)

### Orchestration
- Payment negotiation  
- Invoice settlement  
- Compliance verification  
- Darknet alert creation  
- Optional Q&A step  
- Complete debug trace for judges

### Output
- A full **timeline JSON** consumed by the frontend  
- Includes: invoices, payments, tx_hashes, risk verdicts, and raw API traces

---

# 2. BACKEND ARCHITECTURE

The backend contains 3 major subsystems:

---

## SUBSYSTEM 1 — Locus/x402-Style Compliance Tool Simulator  
**Route:** `POST /locus/compliance/check`

### Behavior:

#### First call per `(txId, role)`:
- Return: **HTTP 402 Payment Required**
- JSON invoice fields:
  - `invoice_id`
  - `price` (0.002 or 0.003)
  - `asset: "USDC"`
  - `receiver_address` (Compliance wallet)
  - `network: "base-testnet"`
  - `tool_name: "ChainGuard Compliance v2"` (optional)

#### Second call with valid `X-402-Payment`:
- Return: **HTTP 200**
- For `role = "sender"` → Clean result  
- For `role = "receiver"` → Darknet hit  
  - `risk_score`
  - `darknet_cluster`
  - `connected_addresses`
  - `recommendation`

#### Internal State:
Track for each `(txId, role)`:
- Whether payment was recorded
- Stored invoice metadata
- Compliance result state

---

## SUBSYSTEM 2 — CDP Server Wallet Integration

Use CDP SDK.

### On startup:
- Create or fetch:
  - **Agent Wallet** (payer)
  - **Compliance Wallet** (invoice receiver)

### Provide helper:
`sendPayment({ amount, asset, to_address, network })`

This MUST:
- Execute a **real testnet on-chain micro-payment**
- Return:
  - `tx_hash`  
  - `from_address`  
  - timestamp  
  - any CDP metadata

---

## SUBSYSTEM 3 — Orchestrator Route (Main Flow)

### Route:
`POST /api/transactions/:id/run-compliance`

This performs the end-to-end agentic workflow.

---

### STEP 1 — Sender initial compliance call
- Call: `/locus/compliance/check?role=sender`
- Expect 402 with invoice
- Timeline event:
  - `type: "call_compliance_agent"`
  - invoice data
  - timestamp
  - `debug.locus_first_call = { request, response }`

---

### STEP 2 — Pay invoice via CDP (sender)
- Extract invoice: price, receiver address
- Send payment via `sendPayment()`
- Capture real `tx_hash`
- Timeline event:
  - `type: "verify_sender"`
  - `amount_paid`
  - `tx_hash`
  - `debug.cdp_payment`

---

### STEP 3 — Retry compliance tool (sender)
- Call again with `X-402-Payment`
- Expect 200 compliance result
- Update sender event with:
  - risk score
  - verdict
  - `debug.locus_second_call`

---

### STEP 4 — Receiver initial compliance call
Same as sender:
- First call → 402 invoice  
- Add timeline event: `verify_receiver (pending)`

---

### STEP 5 — Pay invoice via CDP (receiver)
Same pattern:
- Execute CDP transfer  
- Capture real `tx_hash`  
- Store inside receiver event

---

### STEP 6 — Retry compliance call (receiver)
- Call with x402 header  
- Expect darknet hit result  
- Append to receiver event:
  - `risk_score`
  - `darknet_cluster`
  - `connected_addresses`
  - `recommendation`

---

### STEP 7 — Create darknet alert event
If high risk or cluster present:
- Add:
  - `type: "alert_darknet"`
  - risk_score
  - cluster
  - connected addresses
  - recommendation
  - intel link (placeholder ok)
  - timestamp

---

### STEP 8 — Q&A follow-up (0.00)
Simulate:
- Call compliance tool with Q&A payload  
- Price = 0  
- Add event:
  - `type: "qa_followup"`
  - `price: 0`
  - question + answer

---

# 3. SUCCESS CRITERIA (MUST PASS ALL)

### A. x402 behavior
- First compliance call → **HTTP 402**  
- Retry with payment → **HTTP 200**  
- Sender result: clean  
- Receiver result: darknet hit

### B. Real on-chain payment
- CDP sends **real testnet payment**  
- Returned **tx_hash** is real and verifiable

### C. Complete timeline structure
- All event types appear  
- Full invoice → payment → retry cycle  
- Q&A event  
- Darknet alert event  
- All events include timestamps and debug fields

### D. No frontend hardcoding required
- All data the frontend needs is in the returned JSON

### E. End-to-end run works with a single call:
```
POST /api/transactions/9821/run-compliance
```

### F. Economic correctness
- Sum(invoices) = totalComplianceSpend (0.005 USDC)
