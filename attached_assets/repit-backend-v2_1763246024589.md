
# BACKEND_INSTRUCTIONS_v2.md
### Replit – Backend Specification (Aligned with CDP Agent Wallets, Locus, x402)

---

## 1. PURPOSE

Implement a Node.js/Express backend that:

1. Uses **Coinbase CDP Agent/Server Wallets** to send a **real testnet micro‑payment**.
2. Implements an **x402‑style payment flow**:
   - Initial HTTP request → `402 Payment Required` with PaymentRequirements.
   - Wallet payment.
   - Retry with `X-PAYMENT` header.
   - Optional `X-PAYMENT-RESPONSE` on success.
3. Simulates a **Locus‑discoverable compliance tool** (resource server + inline facilitator).
4. Returns a **rich timeline JSON** that the frontend uses as its single source of truth.

This must be conceptually and terminologically consistent with:

- CDP Agent Wallets / Server Wallet v2.
- x402 protocol.
- Locus as an x402‑native tool marketplace and router.

---

## 2. HIGH‑LEVEL COMPONENTS

Implement three main pieces:

1. **Compliance Tool Simulator** – an x402 resource server (and facilitator in this demo).
2. **CDP Wallet Integration** – Agent Wallet capable of sending micro‑payments.
3. **Orchestrator Route** – Arcot agent calling the tool, paying invoices, and building a timeline.

---

## 3. COMPLIANCE TOOL SIMULATOR (x402 RESOURCE SERVER)

### 3.1 Route

- `POST /tools/compliance/check`

This represents a Locus‑discoverable tool named e.g. `"ChainGuard Compliance v2"`.

### 3.2 Behavior by Request Phase

#### Phase 1: Initial Request (no payment yet)

- Input body includes:
  - `txId`
  - `role` ("sender" or "receiver")
  - Basic transaction metadata.
- If the server has NOT yet seen a valid payment for `(txId, role)`:
  - Respond with **HTTP 402 Payment Required**.
  - Include a JSON body matching a simplified **PaymentRequirements** shape, for example:

    ```json
    {
      "tool": {
        "id": "chain-guard-compliance-v2",
        "name": "ChainGuard Compliance v2"
      },
      "payment_requirements": {
        "amount": "0.002",       // or "0.003" for receiver
        "asset": "USDC",
        "chain_id": "84532",
        "recipient": "<compliance_wallet_address>",
        "expires_at": "<ISO8601>",
        "description": "Sender compliance check"
      }
    }
    ```

  - No `X-PAYMENT-RESPONSE` header is set in this phase.

#### Phase 2: Paid Request (with X-PAYMENT header)

- When the server receives a request for `(txId, role)` with:
  - `X-PAYMENT: <base64-encoded payment payload>`
- It should:
  - Validate (for the demo, basic validation) that this corresponds to a payment of the required `amount` and `asset` to the required `recipient`.
  - In a real deployment this validation would be handled or delegated to an x402 facilitator; in this demo we treat this service as “resource + facilitator”.

- If validation passes:
  - Respond with **HTTP 200 OK**.
  - Include a JSON “compliance result” object:
    - For `role = "sender"`:
      - Low risk score, no darknet cluster.
    - For `role = "receiver"`:
      - High risk score, non‑null darknet cluster, recommendation to flag.
  - Optionally include:
    - `X-PAYMENT-RESPONSE: <base64-encoded response>` header summarizing payment settlement (which can be echoed into the timeline by the orchestrator).

### 3.3 Internal State

- Use simple in‑memory maps to track, per `(txId, role)`:
  - PaymentRequirements (from 402 body).
  - Whether a valid payment has been observed.
  - The final compliance result object.

## 4. CDP WALLET INTEGRATION (AGENT WALLET)

### 4.1 Initialization

Use CDP SDK to:

- Initialize client with:
  - `CDP_API_KEY_ID`
  - `CDP_API_KEY_SECRET`
  - `CDP_WALLET_SECRET`
  - `CDP_PROJECT_ID`
- Create or fetch:
  - **Agent Wallet** (the Arcot agent’s wallet).
- Optionally create a labeled **Compliance Wallet** that matches the `recipient` address used in PaymentRequirements.

### 4.2 Helper: sendPayment

Implement a helper function:

`sendPayment({ amount, asset, to_address, chain_id })`

Responsibilities:

1. Construct a transaction that:
   - Sends `amount` of `asset` (USDC) on the specified chain (e.g. Base Sepolia / chain_id 84532).
   - From the Agent Wallet → `to_address`.
2. Submit the transaction via CDP.
3. Return a structured object:

```json
{
  "tx_hash": "0x...",
  "from": "<agent_wallet_address>",
  "to": "<to_address>",
  "amount": "0.002",
  "asset": "USDC",
  "chain_id": "84532",
  "raw_response": { ... }     // CDP transaction response
}
```

This function underpins the actual x402 payment.

## 5. ORCHESTRATOR ROUTE (ARCOT AGENT)

### 5.1 Route

- `POST /api/transactions/:id/run-compliance`

This should:

1. Accept a transaction ID path param (e.g. `9821`).
2. Either look up the transaction or embed a hardcoded example (8,200 USDC inflow).
3. Execute the full sender + receiver compliance flow.
4. Construct and return the **timeline JSON** for the frontend.

### 5.2 Logical Steps

#### Step A: Sender Phase – Initial Call

- Build a request to `/tools/compliance/check` with:
  - `role: "sender"`
  - tx metadata.
- Capture outgoing request (URL, headers, body).
- Capture response (402 status + PaymentRequirements body).
- Create timeline event:

```json
{
  "type": "call_compliance_agent",
  "role": "sender",
  "status": "completed",
  "ts": "...",
  "invoice": { ...payment_requirements... },
  "debug": {
    "locus_first_call": {
      "request": { "url": "...", "headers": {...}, "body": {...} },
      "response": { "status": 402, "body": {...} }
    }
  }
}
```

#### Step B: Sender Phase – Wallet Payment

- Using PaymentRequirements, call `sendPayment()`:
  - `amount = payment_requirements.amount`
  - `asset = payment_requirements.asset`
  - `to_address = payment_requirements.recipient`
  - `chain_id = payment_requirements.chain_id`
- Record the `tx_hash` and other metadata.
- Construct an x402 payment payload representing this transaction and base64‑encode it to create the `X-PAYMENT` header for the retry.
- Append or create a `verify_sender` event:

```json
{
  "type": "verify_sender",
  "role": "sender",
  "status": "payment_sent",
  "amount_paid": "0.002",
  "asset": "USDC",
  "tx_hash": "0x...",
  "ts": "...",
  "debug": {
    "cdp_payment": {
      "request": {...},
      "response": {...}
    }
  }
}
```

#### Step C: Sender Phase – Retry with X-PAYMENT

- Call `/tools/compliance/check` again with:
  - Same logical body.
  - Header: `X-PAYMENT: <base64-payload>`.
- Capture outgoing request and response (200).
- Extend the `verify_sender` event:

```json
"result": {
  "risk_score": 12,
  "verdict": "approved",
  "notes": ["no darknet hits", "low activity"]
},
"debug": {
  "locus_second_call": {
    "request": {... includes X-PAYMENT ...},
    "response": { "status": 200, "body": {...result...}, "headers": {"X-PAYMENT-RESPONSE": "..."} }
  }
}
```

#### Step D: Receiver Phase – Repeat Steps A–C

Perform the same three phases for `role: "receiver"`, but return a high‑risk/darknet result from the tool.

- `verify_receiver` event:
  - Contains `amount_paid: "0.003"`
  - High `risk_score` (e.g. 85).
  - Non‑null `darknet_cluster` (e.g. `"Silk Road 2.0"`).
  - Recommendation field (e.g. `"flag_for_manual_review"`).

#### Step E: Alert Event

Based on the receiver result:

- Add `alert_darknet` event:

```json
{
  "type": "alert_darknet",
  "status": "alert",
  "ts": "...",
  "risk_score": 85,
  "cluster": "Silk Road 2.0",
  "connected_addresses": 12,
  "recommendation": "Flag this transaction for manual review and SAR filing.",
  "intel_url": "https://example.com/darknet-report/cluster-472"
}
```

#### Step F: Follow‑up Q&A Event (0.00)

- Optionally perform a third tool call with a `question` field.
- Simulate that, by contract, Q&A is price 0.00.
- Add `qa_followup` event:

```json
{
  "type": "qa_followup",
  "status": "completed",
  "ts": "...",
  "price": "0.00",
  "question": "How confident is this darknet classification vs false positive?",
  "answer": "We estimate >97% confidence based on patterns X, Y, Z.",
  "debug": {
    "qa_call": {
      "request": {...},
      "response": {...}
    }
  }
}
```

## 6. RESPONSE FORMAT

The orchestrator route returns:

```json
{
  "txId": 9821,
  "header": {
    "token": "USDC",
    "amount": 8200,
    "from": "0x19a3…b7c2",
    "to": "Arcot Treasury Wallet",
    "status": "aml_flag",
    "risk_score": 87,
    "risk_signals": [
      "new_wallet",
      "new_device",
      "risky_geo_bg",
      "darknet_hit_on_receiver"
    ]
  },
  "timeline": [
    { /* call_compliance_agent (sender) */ },
    { /* verify_sender */ },
    { /* call_compliance_agent (receiver) */ },
    { /* verify_receiver */ },
    { /* alert_darknet */ },
    { /* qa_followup */ }
  ],
  "totalComplianceSpend": "0.005",
  "contract": {
    "name": "ChainGuard Compliance v2",
    "pricing": "x402 metered; Q&A bundled for 24h"
  }
}
```

Requirements:

- Each timeline entry must include a `debug` block with raw HTTP request/response data.
- All numeric values (amounts, risk scores) must be consistent with the economic narrative (e.g., sum of invoices = totalComplianceSpend).

## 7. SUCCESS CRITERIA

The backend is considered correct when all of the following hold:

1. **x402 Behavior**
   - First call to `/tools/compliance/check` for a given `(txId, role)` returns:
     - HTTP 402
     - Body with `payment_requirements`.
   - Paid retry call with `X-PAYMENT` header returns:
     - HTTP 200
     - JSON result.
   - Receiver’s 200 result includes a non‑null darknet cluster and high risk score.

2. **Real On‑Chain Payment via CDP**
   - `sendPayment()` uses CDP to submit a real testnet transaction.
   - The returned `tx_hash` is valid and can be inspected in a block explorer.
   - At least one timeline event (`verify_sender` or `verify_receiver`) contains this `tx_hash`.

3. **Timeline Completeness**
   - Orchestrator returns a `timeline` with sender + receiver flows and derived events (`alert_darknet`, `qa_followup`).
   - Each event carries a `type`, `status`, `ts`, and `debug`.

4. **Economic Consistency**
   - Invoice amounts in PaymentRequirements (e.g., 0.002 + 0.003) sum to `totalComplianceSpend` (0.005).
   - The contract name and pricing fields match the narrative of “Locus tool; x402 metered; Q&A bundled”.

5. **Single Entry Point**
   - A single call to:
     - `POST /api/transactions/9821/run-compliance`
   - Executes the full workflow and returns the final JSON in one response.

6. **No Frontend Hardcoding Needed**
   - All fields the frontend needs (amounts, labels, risk scores, clusters, addresses, header names) are present in this JSON.

## 8. VALIDATION CHECKLIST

Use this to validate behavior before handing off to the frontend or judges.

### 8.1 Compliance Tool

- [ ] `POST /tools/compliance/check` (sender, no `X-PAYMENT`) → 402 with PaymentRequirements JSON.
- [ ] `POST /tools/compliance/check` (sender, with `X-PAYMENT`) → 200 with low‑risk sender result.
- [ ] Same pattern for receiver → 200 with high‑risk, darknet cluster result.

### 8.2 CDP Wallet

- [ ] On server startup, Agent Wallet is created or loaded; address is logged.
- [ ] Manual invocation of `sendPayment()` sends a micro‑payment and returns a valid `tx_hash`.
- [ ] The transaction appears in a testnet explorer for the configured chain.

### 8.3 Orchestrator

- [ ] `POST /api/transactions/9821/run-compliance` returns:
  - `header` with token, amount, from, to, status, risk fields.
  - `timeline` array with the expected types.
  - `totalComplianceSpend` and `contract`.

- [ ] At least one `verify_*` event contains:
  - `amount_paid`
  - `tx_hash`
  - `debug.cdp_payment` with raw CDP request/response.
  - `debug.locus_first_call` and `debug.locus_second_call`.

- [ ] `alert_darknet` event contains:
  - `risk_score`
  - `cluster`
  - `connected_addresses`
  - `recommendation`
  - `intel_url` (even if placeholder).

- [ ] `qa_followup` event contains:
  - `price: "0.00"`
  - `question`
  - `answer`
  - `debug.qa_call`.

### 8.4 JSON Shape

- [ ] All fields present as per this spec.
- [ ] No internal error details or stack traces leak into API responses.
- [ ] Types are consistent (strings vs numbers) in a way the frontend can handle.

## 9. COMPLETION DEFINITION

The backend is **done** when:

- All validation checklist items pass.
- The frontend can render the full Transaction Details view using ONLY this JSON.
- Judges can open “API details” for a payment step and see:
  - A real PaymentRequirements 402 response.
  - A real CDP wallet payment request/response with tx_hash.
  - A real retry request including an `X-PAYMENT` header and 200 result.
